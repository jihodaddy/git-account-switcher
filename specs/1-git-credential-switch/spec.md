# Feature Specification: Git Account Switcher

## Overview

Windows 환경에서 여러 개의 Git 계정(GitHub, GitLab 등)을 손쉽게 전환하며 사용할 수 있는 온디바이스(로컬 실행) 데스크톱 애플리케이션. 사용자는 개인용, 업무용 등 복수의 Git 계정을 등록하고, 저장소별 또는 전역으로 자격증명(credential)을 전환하여 인증 오류 없이 Git 작업을 수행할 수 있다.

## Problem Statement

현재 Windows에서 여러 Git 계정을 사용하려면 수동으로 자격증명 관리자를 편집하거나, `.gitconfig` 파일을 직접 수정하거나, SSH 키를 수동 전환해야 한다. 이 과정은 번거롭고 오류가 발생하기 쉬우며, 잘못된 계정으로 커밋하거나 인증 실패가 반복되는 문제를 야기한다.

## User Scenarios & Testing

### Scenario 1: 계정 등록
- **Given**: 사용자가 앱을 처음 실행한 상태
- **When**: 새 Git 계정 정보(이름, 이메일, 호스트, 인증 토큰/비밀번호)를 입력
- **Then**: 계정이 앱에 저장되고 목록에 표시됨
- **Acceptance**: 최소 2개 이상의 계정을 등록할 수 있어야 함

### Scenario 2: 전역 계정 전환
- **Given**: 2개 이상의 계정이 등록된 상태 (동일 호스트 포함 가능)
- **When**: 사용자가 특정 계정을 "활성 계정"으로 선택
- **Then**: 해당 호스트의 기존 자격증명이 삭제되고 선택한 계정의 자격증명으로 교체됨. Git 전역 설정(user.name, user.email)도 변경됨
- **Acceptance**: 전환 후 `git push`/`git pull` 등의 작업이 인증 오류 없이 수행됨

### Scenario 3: 저장소별 계정 지정
- **Given**: 특정 로컬 저장소 경로가 지정된 상태
- **When**: 사용자가 해당 저장소에 특정 계정을 연결
- **Then**: 해당 저장소의 로컬 Git 설정이 지정된 계정 정보로 구성됨
- **Acceptance**: 전역 설정과 독립적으로 저장소별 계정이 적용됨

### Scenario 4: 현재 활성 계정 확인
- **Given**: 계정이 전환된 상태
- **When**: 사용자가 앱을 열거나 상태를 확인
- **Then**: 현재 활성화된 계정 정보와 연결된 호스트가 명확히 표시됨

### Scenario 5: 계정 삭제
- **Given**: 등록된 계정이 있는 상태
- **When**: 사용자가 특정 계정을 삭제
- **Then**: 해당 계정 정보가 앱과 시스템 자격증명에서 제거됨
- **Acceptance**: 삭제된 계정의 자격증명이 시스템에 남지 않음

## Clarifications

### Session 2026-04-07

- Q: Git Credential Manager(GCM) 충돌 처리 전략은? → A: GCM을 유지하고, 앱이 Windows 자격증명 관리자의 GCM 항목을 직접 갱신하는 방식으로 동작한다.
- Q: 동일 호스트(예: github.com)에 복수 계정이 있을 때 처리 방식은? → A: 전환 시 해당 호스트의 기존 자격증명을 삭제 후 새 계정의 자격증명으로 교체한다.
- Q: 자격증명(토큰) 만료/오류 시 동작은? → A: 계정 전환 시 호스트에 연결하여 토큰 유효성을 검증하고, 실패 시 경고를 표시한다. (네트워크 불가 시에는 검증 없이 전환 허용)

## Functional Requirements

### 필수 (Must Have)

- **FR-1**: 사용자는 Git 계정을 등록할 수 있어야 한다 (이름, 이메일, 호스트 URL, 인증 수단)
- **FR-2**: 사용자는 최소 2개 이상의 Git 계정을 등록하고 관리할 수 있어야 한다
- **FR-3**: 사용자는 전역(global) Git 자격증명을 선택한 계정으로 전환할 수 있어야 한다
- **FR-4**: 계정 전환 시 Git Credential Manager(GCM)가 사용하는 Windows 자격증명 관리자 항목을 직접 갱신하여, GCM을 비활성화하지 않고 공존해야 한다
- **FR-5**: 계정 전환 시 Git 전역 설정(user.name, user.email)이 자동으로 변경되어야 한다
- **FR-6**: 사용자는 현재 활성화된 계정을 확인할 수 있어야 한다
- **FR-7**: 앱은 인터넷 연결 없이 온디바이스(로컬)에서 실행 가능해야 한다
- **FR-8**: 저장된 자격증명(토큰, 비밀번호)은 안전하게 암호화되어 저장되어야 한다
- **FR-9**: 사용자는 등록된 계정을 수정하거나 삭제할 수 있어야 한다

- **FR-10**: 계정 전환 시 해당 호스트에 연결하여 토큰 유효성을 검증해야 한다. 유효하지 않은 경우 사용자에게 경고를 표시하되, 전환 자체는 허용한다
- **FR-11**: 네트워크 연결이 불가능한 경우 유효성 검증을 건너뛰고 전환을 허용해야 한다

### 선택 (Should Have)

- **FR-12**: 사용자는 특정 로컬 저장소에 개별 계정을 연결할 수 있어야 한다 (저장소별 설정)
- **FR-13**: 사용자는 GitHub, GitLab, Bitbucket 등 주요 Git 호스트를 선택할 수 있어야 한다
- **FR-14**: 시스템 트레이에서 빠른 계정 전환이 가능해야 한다

## Success Criteria

1. 사용자가 3번의 클릭 이내로 활성 Git 계정을 전환할 수 있다
2. 계정 전환 후 Git 원격 작업(push, pull, clone)이 인증 오류 없이 성공한다
3. 최소 5개 이상의 서로 다른 Git 계정을 동시에 관리할 수 있다
4. 앱이 인터넷 연결 없이 완전히 동작한다 (자격증명 전환 기능)
5. 사용자가 계정을 등록하고 첫 전환을 완료하기까지 5분 이내에 수행할 수 있다
6. 저장된 자격증명이 앱 외부에서 평문으로 접근 불가능하다

## Key Entities

### Account (계정)
- 표시 이름 (Display Name)
- Git 사용자 이름 (user.name)
- Git 이메일 (user.email)
- 호스트 URL (예: github.com, gitlab.com)
- 인증 수단 (Personal Access Token 또는 비밀번호)
- 활성 상태 (Active/Inactive)

### Repository Binding (저장소 연결)
- 로컬 저장소 경로
- 연결된 계정
- 설정 범위 (전역/로컬)

## Scope & Boundaries

### In Scope
- Windows OS 전용 데스크톱 애플리케이션
- HTTPS 기반 Git 자격증명 전환
- Windows 자격증명 관리자 통합
- Git 전역/로컬 설정 자동 변경
- 계정 정보의 암호화된 로컬 저장

### Out of Scope
- macOS/Linux 지원
- SSH 키 기반 인증 전환 (향후 확장 가능)
- Git GUI 기능 (커밋, 브랜치 관리 등)
- Git 호스팅 서비스 API 연동 (계정 유효성 검증 등)
- 클라우드 동기화

## Dependencies & Assumptions

### Dependencies
- Windows 운영 체제 (Windows 10 이상)
- Git이 시스템에 설치되어 있어야 함
- Windows 자격증명 관리자 접근 권한

### Assumptions
- 사용자는 이미 Git 호스트에 계정을 가지고 있으며, 인증 토큰을 생성하는 방법을 알고 있다
- HTTPS 방식의 Git 원격 저장소를 사용한다고 가정한다
- 사용자에게 Windows 자격증명 관리자에 대한 읽기/쓰기 권한이 있다
- Git이 PATH에 등록되어 명령줄에서 접근 가능하다

## Risks

- **자격증명 보안**: 토큰/비밀번호를 로컬에 저장하므로, 암호화가 미흡할 경우 보안 취약점이 발생할 수 있음
- **Windows 자격증명 관리자 호환성**: Windows 버전별로 자격증명 관리자 API 동작이 다를 수 있음
- **Git Credential Helper 충돌**: GCM을 유지하며 자격증명 관리자 항목을 직접 갱신하는 방식으로 공존하되, GCM이 자격증명을 캐시/덮어쓰는 타이밍에 주의 필요
